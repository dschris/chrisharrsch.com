Architecture Overview

This setup exposes self-hosted applications running on an server to the public internet via custom subdomains, with automatic dynamic DNS updates, SSL termination, and optional user authentication via Clerk.

Traffic flow: Public internet → Cloudflare edge (proxy + DDoS shield) → Home router (port forward) → Nginx Proxy Manager (NPM) → Individual app containers.

Key design decisions:

  • A single wildcard A record (*.yourdomain.com) in Cloudflare points to your home IP. This means you never need to touch Cloudflare DNS when adding a new app — NPM is the only place routing is managed.

  • NPM runs in Host network mode so it can bind directly to ports 80 and 443. This requires server's own web UI to be moved off those ports first.

  • The Cloudflare DDNS container maintains the wildcard A record automatically as your home IP changes. It makes outbound API calls only and uses no ports.

  • SSL is terminated at NPM using a *.yourdomain.com Let's Encrypt wildcard certificate issued via DNS challenge (not HTTP challenge), so no port 80 exposure is required for cert issuance.

Note on local access: Accessing your public subdomains from inside your own network may not work depending on your ISP gateway and router configuration. The two most common causes are ISP-level port blocking (affects all connections) and hairpin NAT limitations on consumer routers (affects local connections only). See the Troubleshooting Reference and Operational Notes sections for details and workarounds.

Prerequisites

Before starting, confirm the following:

  • You own a domain name registered with any registrar (e.g. Namecheap, GoDaddy, Google Domains)

  • You have a Cloudflare account (free tier is sufficient)

  • Your server has Nginx Proxy Manager available to install as a Docker container

  • Your router supports port forwarding

  • Your ISP does not block inbound ports 80 and 443 at the gateway level. Check your ISP modem or gateway admin page before starting — many ISPs block these ports by default and must be explicitly opened in the ISP equipment's interface, separate from your router's port forwarding rules.

  • You have no static public IP (hence the need for DDNS)

Phase 1: Move DNS to Cloudflare

Your current registrar remains where your domain is registered, but Cloudflare becomes your authoritative DNS provider. This is required for the DDNS updater and gives you the Cloudflare proxy as a free DDoS/IP-hiding layer.
  1. Add your domain to Cloudflare: Log in to dash.cloudflare.com → Add a site → enter your domain name → choose the Free plan.
  2. Cloudflare will scan and import your existing DNS records. Review them carefully and keep any records for services you are already running (e.g. a hosted website, email records, or other subdomains).
  3. Cloudflare will give you two nameservers (e.g. nina.ns.cloudflare.com and otto.ns.cloudflare.com). Copy these.
  4. Point your registrar to Cloudflare nameservers: Log in to your domain registrar, find the nameserver settings for your domain, switch to custom nameservers, and enter both Cloudflare NS values. Save.
  5. Propagation takes up to 24 hours but is usually under an hour.
  6. Verify existing records are intact: Once Cloudflare is active, confirm that any records for services you were already running (apex domain, www, email MX records, etc.) are present and correctly configured. If an existing service breaks after switching, check whether its DNS records need to be set to DNS-only (gray cloud) rather than proxied — some hosting providers require direct DNS resolution and do not work behind the Cloudflare proxy.

Phase 2: Set Up Wildcard DNS for Your Homelab

Rather than creating individual A records for each app subdomain, use a single wildcard A record pointing to your home IP. This means you only ever manage routing in NPM — Cloudflare DNS becomes set-and-forget.
  1. Check for an existing wildcard record: Cloudflare may have imported a * A record pointing to a third-party hosting provider's IPs. Before deleting it, confirm that your apex domain and any important subdomains have their own explicit records in Cloudflare. Once a wildcard is deleted, only subdomains with explicit records will continue to resolve.
  2. Delete any existing wildcard A record: In Cloudflare → DNS → Records, delete any existing * A records you no longer need.
  3. Add a homelab wildcard A record: In Cloudflare → DNS → Records → Add record: Type A, Name *, IPv4 address set to your current public IP (find it at whatismyip.com), Proxy status Proxied (orange cloud), TTL Auto → Save.
  4. Note: Cloudflare always prefers the most specific match. Any explicit subdomain records will still win over this wildcard. The wildcard only catches subdomains with no explicit record of their own.
  5. Why proxied? With the orange cloud on, external clients hit Cloudflare's edge rather than your home IP directly. This hides your home IP and provides DDoS protection. The trade-off is that only standard ports 80 and 443 can be proxied.

Phase 3: Create a Cloudflare API Token

This single token is used by both the DDNS container (to update the wildcard A record) and Nginx Proxy Manager (to issue SSL certificates via DNS challenge). One token covers both use cases.
  1. In Cloudflare → top-right avatar → My Profile → API Tokens → Create Token.
  2. Use the Edit zone DNS template.
  3. Under Zone Resources, set it to Specific zone → select your domain.
  4. Click Continue to summary → Create Token.
  5. Copy and save this token immediately — it is only shown once. Store it in your password manager.
  6. Your Zone ID is on the right-hand sidebar of your domain's Overview page in Cloudflare. You will need this for the DDNS config.json in Phase 6.

Phase 4: Prepare the Server for NPM in Host Mode

NPM must run in Host network mode so it can bind directly to ports 80 and 443. server's own web UI sits on those ports by default, so you must move it first.
  1. In Server dashboard → Settings → Management Access.
  2. Change the HTTP port from 80 to 8080.
  3. Change the HTTPS port from 443 to 8443.
  4. Save. The server will restart its web UI. Going forward, access the server dashboard by appending :8080 to your server's LAN IP (e.g. http://YOUR_SERVER_IP:8080).
  5. Why Host mode? In bridge mode, NPM binds only to its internal container network rather than the host's physical network interfaces, causing it to listen on localhost only and refuse all connections arriving from the router's port forward. Host mode makes NPM bind directly to all of the host's network interfaces.

Phase 5: Deploy Nginx Proxy Manager

Install NPM as a Docker container in Host network mode with persistent data volumes.
  1. In Server dashboard → Docker tab → Add Container (or search Community Applications for 'Nginx Proxy Manager').
  2. Set Network Type to Host.
  3. Add the following volume mappings to preserve your configuration across container rebuilds:
  4.   • Host path /mnt/user/appdata/nginx-proxy-manager/data → Container path /data
  5.   • Host path /mnt/user/appdata/nginx-proxy-manager/letsencrypt → Container path /etc/letsencrypt
  6. Do not add any port mappings — in Host mode the container shares the host's network stack directly and port mappings are not used.
  7. Click Apply and start the container.
  8. Access the NPM web UI at http://YOUR_SERVER_IP:81. Port 81 is NPM's admin interface.
  9. Default login credentials: Email admin@example.com, Password changeme. NPM will force you to change both on first login.
  10. Verify NPM is binding correctly by running the following in the server terminal. You should see 0.0.0.0:80, 0.0.0.0:443, and 0.0.0.0:81 in the output:
  11.   netstat -tlnp | grep -E ':80|:81|:443'
  12. If port 443 shows as bound to 127.0.0.1 rather than 0.0.0.0, NPM is not correctly in Host mode or a port conflict exists. See the Troubleshooting Reference section.
  13. Tip: Once NPM is working, create a proxy host for NPM itself pointing to localhost:81 under a subdomain (e.g. npm.yourdomain.com). This lets you access the admin UI over HTTPS and allows you to block external access to raw port 81.

Phase 6: Deploy the Cloudflare DDNS Container

This container polls your public IP and updates the wildcard A record in Cloudflare whenever it changes. It makes outbound API calls only and uses no ports.
  1. Create the config file. SSH into your server (or use the server terminal) and run:
  2.   mkdir -p /mnt/user/appdata/cloudflare-ddns
  3.   nano /mnt/user/appdata/cloudflare-ddns/config.json
  4. Paste the following, replacing the placeholder values with your API token (from Phase 3) and Zone ID (from your domain's Cloudflare Overview page):
  5. config.json 
    {
  "cloudflare": [
    {
      "authentication": { "api_token": "YOUR_API_TOKEN_HERE" },
      "zone_id": "YOUR_ZONE_ID_HERE",
      "subdomains": [ { "name": "*", "proxied": true } ]
    }
  ],
  "a": true,
  "aaaa": false,
  "purgeUnknownRecords": false,
  "ttl": 300
}
  6. Add the container → Docker → Add Container (or search Community Applications for 'cloudflare-ddns' by timothymiller):
  7.   • Repository: timothyjmiller/cloudflare-ddns:latest
  8.   • Network Type: Host
  9.   • Path mapping: Host /mnt/user/appdata/cloudflare-ddns/config.json → Container /config.json
  10. Click Apply. Open the container log — a successful first run shows: Updating A record for *.yourdomain.com → [your public IP].
  11. The container re-checks every 300 seconds and updates Cloudflare only when your IP has changed.

Phase 7: Router Port Forwarding

Cloudflare's proxy forwards traffic to your public IP on ports 80 and 443. Your router needs to send that traffic to the server running NPM.
  1. Log into your router admin panel.
  2. Create two port forward rules:
  3.   • External port 80 → your server's LAN IP, internal port 80
  4.   • External port 443 → your server's LAN IP, internal port 443
  5. Ensure your server has a static LAN IP so these rules don't break if the server reboots. Set a DHCP reservation by MAC address in your router, or configure a static IP under Settings → Network Settings.

Phase 8: Configure Nginx Proxy Manager

Issue a wildcard SSL certificate and create proxy host entries for each app. Adding a new app in the future only requires adding a proxy host in NPM — no Cloudflare or DDNS changes needed.
  1. Issue a wildcard SSL certificate via DNS challenge (not HTTP challenge — DNS challenge is required for wildcard certs and does not require port 80 to be reachable for cert issuance):
  2.   • In NPM → SSL Certificates → Add SSL Certificate → Let's Encrypt.
  3.   • Domain Names: *.yourdomain.com
  4.   • Toggle Use a DNS Challenge → on.
  5.   • DNS Provider: Cloudflare.
  6.   • Credentials File Content: dns_cloudflare_api_token=YOUR_API_TOKEN_HERE (same token from Phase 3).
  7.   • Enter your email, agree to the Let's Encrypt TOS → Save.
  8. NPM will place a temporary TXT record in Cloudflare to prove domain ownership, then issue the cert. This takes 60–90 seconds and happens synchronously — if it fails you will see an error on screen. Successfully landing on the certs list means it worked.
  9. The cert will show as 'Not Used' until you attach it to a proxy host. This is normal and just means no proxy host has been assigned the cert yet. Confirm the cert issued correctly by checking that an expiry date approximately 90 days out is shown next to it.
  10. Add a proxy host for each app:
  11.   • Hosts → Proxy Hosts → Add Proxy Host.
  12.   • Domain Names: the full subdomain for this app (e.g. appname.yourdomain.com).
  13.   • Scheme: http — NPM talks to the backend container over HTTP internally and handles HTTPS termination itself. Do not set this to https unless your backend container is itself serving HTTPS.
  14.   • Forward Hostname / IP: the app container's LAN IP or Docker container name if on the same bridge network.
  15.   • Forward Port: the app's internal port.
  16.   • Toggle Block Common Exploits → on.
  17.   • Toggle Websockets Support → on if the app requires it.
  18.   • SSL tab → select the *.yourdomain.com cert → toggle Force SSL and HTTP/2 Support on → Save.
  19. Set Cloudflare SSL mode to Full: In Cloudflare → SSL/TLS → Overview, set the mode to Full (not Flexible, not Full Strict). Flexible causes redirect loops because NPM redirects HTTP to HTTPS while Cloudflare is still sending HTTP to the origin. Full requires encryption to the origin but accepts NPM's Let's Encrypt cert without issue.
  20. If you see ERR_TOO_MANY_REDIRECTS after setup, this is almost always the cause — switch Cloudflare from Flexible to Full.
  21. Access Control Lists (optional): For apps you want to restrict to known IPs without full Clerk auth: NPM → Access Lists → Add Access List → add your trusted IP ranges as allow, set default action to deny → attach to the relevant proxy host under its Details tab.

Phase 9: Clerk Auth for Demo Apps (Optional)

For hosted demo applications where you want to force user registration with an allowlist, Clerk sits in front of the app as an authentication layer. NPM still terminates TLS and routes traffic — Clerk enforces session validation before content is served.
  1. Pattern A — Clerk in the app (preferred for apps you control):
  2.   • Sign up at clerk.com → create an application.
  3.   • Install the Clerk SDK: npm install @clerk/nextjs (Next.js) or npm install @clerk/clerk-react @clerk/clerk-sdk-node (React + Express).
  4.   • Wrap your app with <ClerkProvider> and protect routes with auth() middleware.
  5.   • In Clerk Dashboard → Users → manually invite or allowlist specific email addresses or domains.
  6.   • Your NPM proxy host for this app remains unchanged — Clerk auth happens at the application layer.
  7. Pattern B — Auth middleware proxy (for third-party containers you don't control):
  8.   • Add oauth2-proxy as a container on your server.
  9.   • Configure it with Clerk's OIDC endpoints (available at https://YOUR_CLERK_DOMAIN/.well-known/openid-configuration).
  10.   • In NPM, point the proxy host for that app at oauth2-proxy's port rather than the app directly.
  11.   • oauth2-proxy handles the login redirect → Clerk handles auth → oauth2-proxy forwards the validated session to the upstream app.
  12. Clerk allowlisting (applies to both patterns):
  13.   • In Clerk Dashboard → User & Authentication → Restrictions → enable Allowlist mode.
  14.   • Add specific email addresses or whole domains you want to permit.
  15.   • Disable public sign-up so only allowlisted users can register.

Phase 10: Verification Checklist

Run through these after initial setup:

  ☐ dig appname.yourdomain.com returns a Cloudflare IP (not your home IP) — confirms the proxy is active

  ☐ https://appname.yourdomain.com loads from an external network with a valid *.yourdomain.com cert

  ☐ DDNS container log shows a successful A record update on first run

  ☐ After manually changing the Cloudflare wildcard A record IP, the DDNS container corrects it within 5 minutes

  ☐ NPM access list blocks an IP not on your allowlist with a 403

  ☐ Clerk-gated app redirects unauthenticated users to the Clerk sign-in page

  ☐ Your existing domain (apex and www) still resolves correctly — Cloudflare did not break those records

  ☐ No ERR_TOO_MANY_REDIRECTS — Cloudflare SSL mode is set to Full

Operational Notes

Adding a new app: Add one proxy host in NPM (subdomain + forward IP/port + attach the existing *.yourdomain.com cert). That's it. No Cloudflare changes, no DDNS config changes.

Local access: Accessing your public subdomains from inside your own network may fail depending on your router and ISP gateway. The most common causes are ISP-level port blocking (see Troubleshooting) and hairpin NAT — where the router drops connections destined for its own external IP originating from inside the network. Options for local access include:

  • Access apps directly via their LAN IP and port (e.g. http://YOUR_SERVER_IP:APP_PORT)

  • Add a DNS rewrite in a local DNS resolver (AdGuard Home, Pi-hole) mapping *.yourdomain.com → your server's LAN IP

  • Add a hosts file entry on each device: YOUR_SERVER_IP appname.yourdomain.com

Cloudflare caching: For APIs or apps returning dynamic data, add a Cloudflare Page Rule or Transform Rule to bypass caching for those paths, or set Cache Level: Bypass on the relevant domain.

NPM admin UI: Access at http://YOUR_SERVER_IP:81. In Host mode there is no separate container IP — the container shares the host's network stack directly.

Server dashboard: After moving management ports in Phase 4, access the server dashboard at http://YOUR_SERVER_IP:8080.

Troubleshooting Reference

Cloudflare 521 (Connection refused): NPM is not listening on port 443. Verify NPM is running (docker ps | grep proxy) and that netstat -tlnp | grep 443 shows 0.0.0.0:443 bound to nginx, not just 127.0.0.1:443. If bound to localhost only, NPM is not correctly in Host mode or a port conflict exists — confirm server's management ports were moved off 80/443 in Phase 4.

Cloudflare 522 (Connection timed out): Cloudflare reached your public IP but the connection timed out. Work through these in order: (1) Check your ISP modem or gateway config page for port blocking — many ISPs block inbound ports 80 and/or 443 by default at the gateway level, upstream of your router entirely. This must be disabled in the ISP equipment's admin interface, not your router. (2) Check port forwarding rules on your router. (3) Confirm NPM is bound to 0.0.0.0:443 not localhost. (4) Check that no firewall on the server is blocking inbound 443.

ERR_TOO_MANY_REDIRECTS: Cloudflare SSL mode is set to Flexible. Change it to Full in Cloudflare → SSL/TLS → Overview.

NPM bound to 127.0.0.1 only: server's own nginx is likely still holding ports 80 and 443. Confirm server management ports are set to 8080/8443 (Phase 4), then restart the NPM container.

Cert shows 'Not Used' in NPM: Normal — this just means no proxy host has been assigned the cert yet. Confirm the cert issued correctly by checking that an expiry date approximately 90 days out is shown.

522 on all connections including local: Check your ISP gateway for port blocking first — ISP-level port blocks affect all connections. If external users are also getting 522, the ISP gateway is the most likely cause. If external users are unaffected but local access fails, this is likely hairpin NAT — see Local Access in Operational Notes above.

Architecture Overview

This setup provides network-wide ad-blocking and custom DNS resolution to all roaming devices connected to a Tailscale mesh network.

The Design Choice: We use Host-Level Subnet Routing rather than a Tailscale Docker Sidecar.

Why: When assigning a Docker container a custom br0 (MACVLAN) IP on Unraid, Docker's virtual network adapter often fails to correctly route Tailscale's internal IPv6 MagicDNS traffic, causing infinite DNS loops or timeouts on Windows clients. Routing the tunnel directly through the Unraid host's native network stack bypasses this restriction entirely.

Phase 1: Unraid Host Preparation

The most critical step. By default, Unraid isolates the host server from talking to containers on custom networks. We must disable this isolation so Tailscale (running on the host) can reach the Pi-hole (running on br0).
  1. Navigate to your Unraid dashboard and go to Settings > Docker.
  2. Toggle Enable Docker to No and click Apply (you must stop the service to edit core settings).
  3. Switch the UI to Advanced View (toggle in the top right).
  4. Locate Host access to custom networks and change it to Enabled.
  5. Toggle Enable Docker back to Yes and click Apply.

Phase 2: Host-Level Tailscale Configuration

We need to configure the Unraid host's Tailscale node to advertise the subnet where the Pi-hole will live.
  1. Open the Unraid Terminal.
  2. Execute the Tailscale up command, specifically appending the --advertise-routes flag for your br0 subnet.
  3. Command: tailscale up --advertise-routes=10.0.10.2/232
  4. (Note: You can make this more permissive to include more IPs in the range using /24 if preferred, but /32 covers just the single IP). Multiple single IPs can be advertised by using --advertise-routes=10.0.10.2/32,10.0.10.3/32

Phase 3: Pi-hole Docker Deployment

Deploying the DNS server. This container remains purely local and does not spin up its own Tailscale instance.
  1. In Unraid, go to the Docker tab and add/edit the Pi-hole container.
  2. Apply the following core network configurations:
  3.   • Network Type: Custom : br0
  4.   • Fixed IP address: 10.0.10.2
  5.   • Use Tailscale (Sidecar): OFF
  6. Ensure ports 53 (TCP/UDP) and 80 (TCP) are mapped correctly.
  7. Click Apply to build and start the container.
  8. Open the Pi-hole Web UI (http://10.0.10.2/admin). (Note: You may need to set the password by opening a console to the container and running pihole setpassword [PASSWORD]).
  9. Go to Settings > DNS.
  10. Under Interface settings, select Permit all origins. (Required because queries will be arriving from 100.x Tailscale IPs, not just the local 10.0 subnet).
  11. Ensure your Upstream DNS Servers are set strictly to public providers (e.g., Cloudflare 1.1.1.1, Google 8.8.8.8). Do not point upstream to your local router.

Phase 4: Tailscale Admin Console

Connecting the mesh network to the newly advertised route and enforcing the DNS policy.
  1. Log into the Tailscale Admin Console on the web.
  2. Approve the Route:
  3.   • Go to the Machines tab and locate your Unraid server.
  4.   • Click the three-dot menu and select Edit route settings.
  5.   • Toggle the 10.0.10.2/32 subnet route to ON.
  6. Enforce Global DNS:
  7.   • Navigate to the DNS tab.
  8.   • Under Nameservers, click Add nameserver > Custom.
  9.   • Enter the exact local IPv4 address of the Pi-hole: 10.0.10.2.
  10.   • Toggle Override local DNS to ON.
  11.   • Ensure MagicDNS is enabled.

Phase 5: Client-Side Troubleshooting (Windows)

Windows OS has an aggressive tendency to prefer local IPv6 DNS servers (provided by local Wi-Fi routers) over the Tailscale IPv4 tunnel, which can cause DNS timeouts even when the architecture is configured correctly.
  1. If a Windows machine refuses to resolve hostnames after completing Phases 1-4:
  2. Flush the OS Cache:
  3.   • Open an Administrator Command Prompt and run: ipconfig /flushdns
  4.   • Check the DNS server you are using by running nslookup
  5.   • Run ipconfig /release and ipconfig /renew to force the OS to pick up the new DNS settings.
  6.   • Restart the Tailscale client app.
